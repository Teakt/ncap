Épisode 1 — Le Secret sans étiquette. Premier test avec clé : 401 « Unauthorized » — différent du « No API key found » initial : la clé était vue mais reconnue de personne. Cause : le format de credential que je t'avais donné (kongCredType) datait de l'ancienne génération du controller ; KIC 3.x exige un label konghq.com/credential=key-auth sur le Secret. Leçons : lire le delta entre deux messages d'erreur (il localise le maillon cassé), et vérifier la doc de sa version — les contrats des controllers changent entre majeures. (Mea culpa inclus.)
Épisode 2 — Le consumer en double. L'upgrade suivant : le webhook d'admission refuse — « consumer already exists ». Un renommage de ressource entre deux upgrades avait laissé un doublon : pour Helm, renommer = supprimer + créer, et la création précède. Leçons : chaque metadata.name est une ressource à part entière avec son cycle de vie ; et découverte du webhook d'admission — un contrôle qualité posté à l'entrée de l'API Kubernetes.
Épisode 3 — Le champ immuable. Pour ajouter le rate-limiting, tu as recyclé le template key-auth en changeant son plugin: → « The plugin field is immutable ». Un plugin ne change pas de nature ; il fallait un nouveau fichier, une nouvelle ressource. Leçon jumelle : restaurer un fichier avec git checkout --, pas de mémoire — ta restauration manuelle a introduit deux régressions (keyauth sans tiret, mauvais chemin de values).
Épisode 4 — Le nom vide. « resource name may not be empty » : le template référençait .Values.kong.keyauth, la valeur s'appelait kong.auth. Helm ne signale pas un chemin de values inexistant — il rend une chaîne vide, silencieusement. Leçon capitale : helm template avant tout upgrade douteux — le YAML rendu ne ment jamais. Et en corollaire : un intrus était né dans le mauvais namespace (values kong.namespace: kong détourné), à purger à la main.
Épisode 5 — Le tiret. Le webhook encore : « plugin 'keyauth' not enabled ». Le nom officiel est key-auth. Tu avais appliqué une correction sur les trois listées. Leçon de process, pas de technique : une review se traite comme une checklist — chaque point coché ou contesté, jamais sauté. C'est le fil rouge de tes trois phases.
Épisode 6 — L'annotation détournée (le boss final). Release enfin deployed, et pourtant : 401 persistant, consumer PROGRAMMED: Unknown. Les hypothèses successives (label absent, plugin cassé bloquant le push atomique) sont tombées une à une par le constat — et c'est le describe lu ligne à ligne qui a montré : kubernetes.io/ingress.class: ncap. En templatisant, tu avais remplacé kong par la mauvaise valeur — et le consumer attendait pour toujours un controller nommé « ncap ». Leçon : templatiser n'est pas substituer au hasard — chaque remplacement doit préserver le sens de la valeur. Et son Status le disait depuis le début : « Waiting for controller ».
